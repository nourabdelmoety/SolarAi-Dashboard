1 · System architecture overview

Your system has five layers. Keep them separate — this is what makes it scalable later.

Solar sensor (SCT-013) Battery BMS (UART) Thermal probe (DS18B20) Grid meter (PZEM-004T) Irradiance (BH1750) Ambient (DHT22)
↓ serial / I2C / digital
ESP32 (edge device) — reads sensors, publishes MQTT every 15s
↓ MQTT over WiFi (port 1883)
Mosquitto MQTT broker → Node-RED (flow processor) → InfluxDB (time-series DB)
↓ HTTP REST / InfluxDB query
Python AI controller — runs every 15 min, reads DB + APIs, writes decision
↓ MQTT command topic / relay GPIO
Grafana dashboard or Custom HTML dashboard (provided)
ℹ For the prototype, everything except the ESP32 runs on your laptop or a Raspberry Pi 4. When you scale up, you move the backend stack to a small server or cloud VM — the code changes nothing.

2 · Phase 1 — Hardware & sensors

Phase 1 · estimated time: 1–2 weeks

Buy, wire up, and verify your 6 sensors before writing any software.

01
Buy an ESP32 development board
hardware · ~€8

Use the ESP32 DevKit v1 (30-pin) or Wemos D1 Mini ESP32. Both are widely available on Amazon.de, AZ-Delivery, or Berrybase. The ESP32 has built-in WiFi, enough GPIO pins for all 6 sensors, and a large community.

⚠ Do NOT use an Arduino Uno — it has no WiFi and not enough memory. ESP32 only.
02
Wire up your 6 sensors
hardware · ~€35 total for all sensors

SCT-013-030Solar / inverter current outputAnalog (ADC)GPIO34 (ADC1)~€6
PZEM-004T v3Grid bidirectional energy meterUART (Serial2)GPIO16/17~€9
DS18B20Thermal store water temperature1-WireGPIO4~€3
BH1750Solar irradiance (lux → W/m²)I2CSDA=21, SCL=22~€3

All of the above use standard 3.3V logic — directly compatible with ESP32. The SCT-013 needs a 10Ω burden resistor and a voltage divider to bring the signal to ESP32's 0–3.3V ADC range.

3 · Phase 2 — ESP32 firmware

Phase 2 · estimated time: 1 week

Write Arduino-style C++ that reads all sensors and publishes their values over WiFi using MQTT.

03
Install Arduino IDE + ESP32 board + libraries
software setup · free
  1. Download Arduino IDE 2.x from arduino.cc
  2. In Arduino IDE → Preferences → Additional Boards Manager URLs, add: https://raw.githubusercontent.com/espressif/arduino-esp32/gh-pages/package_esp32_index.json
  3. Tools → Board → Boards Manager → search "esp32" → install by Espressif Systems
  4. Install these libraries via Tools → Manage Libraries:
    • PubSubClient by Nick O'Leary (MQTT)
    • ArduinoJson by Benoit Blanchon
    • OneWire + DallasTemperature (DS18B20)
    • BH1750 by Christopher Laws
    • DHT sensor library by Adafruit
    • PZEM-004T-v30 by Jakub Mandula
04
Flash the firmware
core code structure · C++

Here is the core structure of your ESP32 firmware. This is what you flash and it handles all 6 sensors:

// SolarAI ESP32 Firmware — main.cpp
// Reads 6 sensors every 15 seconds, publishes to MQTT broker

#include <WiFi.h>
#include <PubSubClient.h>
#include <ArduinoJson.h>
#include <OneWire.h>
#include <DallasTemperature.h>
#include <BH1750.h>
#include <DHT.h>
#include <PZEM004Tv30.h>

// ── CONFIG ──────────────────────────────────
const char* WIFI_SSID     = "YourWiFiName";
const char* WIFI_PASS     = "YourWiFiPassword";
const char* MQTT_BROKER   = "192.168.1.100"; // IP of your laptop/Pi
const char* MQTT_TOPIC    = "solar/sensors";
const char* MQTT_CMD      = "solar/command"; // receives AI decisions
const int   PUBLISH_EVERY = 15000;           // 15 seconds

// ── SENSOR SETUP ────────────────────────────
OneWire           oneWire(4);
DallasTemperature thermalSensor(&oneWire);
BH1750            lightMeter;
DHT               dht(5, DHT22);
PZEM004Tv30       pzem(Serial2, 16, 17);

WiFiClient        espClient;
PubSubClient      mqttClient(espClient);

void setup() {
  Serial.begin(115200);
  Wire.begin();             // I2C for BH1750
  thermalSensor.begin();
  lightMeter.begin();
  dht.begin();
  connectWiFi();
  mqttClient.setServer(MQTT_BROKER, 1883);
  mqttClient.setCallback(onCommand); // handle AI commands
}

void loop() {
  if (!mqttClient.connected()) reconnectMQTT();
  mqttClient.loop();

  static unsigned long lastPublish = 0;
  if (millis() - lastPublish >= PUBLISH_EVERY) {
    publishSensorData();
    lastPublish = millis();
  }
}

void publishSensorData() {
  thermalSensor.requestTemperatures();
  float thermal_temp = thermalSensor.getTempCByIndex(0);
  float lux          = lightMeter.readLightLevel();
  float irradiance   = lux / 120.0; // approx W/m²
  float ambient_temp = dht.readTemperature();
  float grid_power   = pzem.power();
  float grid_voltage = pzem.voltage();
  // SCT-013 solar current — analog read + convert
  float solar_current = readSCT013(34);
  float solar_kw      = (solar_current * grid_voltage) / 1000.0;

  StaticJsonDocument<256> doc;
  doc["solar_kw"]      = solar_kw;
  doc["thermal_temp"]  = thermal_temp;
  doc["irradiance"]    = irradiance;
  doc["ambient_temp"]  = ambient_temp;
  doc["grid_power_w"]  = grid_power;
  doc["timestamp"]     = millis();
  // battery_soc would come from your BMS UART here

  char buffer[256];
  serializeJson(doc, buffer);
  mqttClient.publish(MQTT_TOPIC, buffer);
}
✓ The AI controller receives data from solar/sensors and publishes commands (SELL / STORE / USE) back to solar/command. The ESP32 subscribes to that topic and activates relay outputs accordingly.

4 · Phase 3 — Backend stack

Phase 3 · estimated time: 2–3 days

Set up Mosquitto + Node-RED + InfluxDB on your laptop. Use Docker to do it in one command.

05
Install Docker Desktop
free · docker.com/products/docker-desktop

Download Docker Desktop for Windows or Mac. This is a container runtime — it lets you run Mosquitto, InfluxDB, and Node-RED as isolated services without installing them directly. For Windows, enable WSL2 when prompted.

06
Run the full stack with one Docker command
create a file called docker-compose.yml in any folder, then run it
# docker-compose.yml — SolarAI backend stack
version: '3.8'
services:

  mosquitto:         # MQTT broker
    image: eclipse-mosquitto:2
    ports: ["1883:1883"]
    volumes:
      - ./mosquitto.conf:/mosquitto/config/mosquitto.conf

  influxdb:          # time-series database
    image: influxdb:2.7
    ports: ["8086:8086"]
    environment:
      DOCKER_INFLUXDB_INIT_MODE: setup
      DOCKER_INFLUXDB_INIT_USERNAME: admin
      DOCKER_INFLUXDB_INIT_PASSWORD: solar1234
      DOCKER_INFLUXDB_INIT_ORG: solarai
      DOCKER_INFLUXDB_INIT_BUCKET: energy_data

  nodered:           # visual flow processor
    image: nodered/node-red:latest
    ports: ["1880:1880"]
    depends_on: [mosquitto, influxdb]

  grafana:           # optional: professional dashboard
    image: grafana/grafana:latest
    ports: ["3000:3000"]
    depends_on: [influxdb]

Also create mosquitto.conf in the same folder:

listener 1883
allow_anonymous true

Then run: docker-compose up -d

After 30 seconds: Node-RED is at http://localhost:1880 · InfluxDB at http://localhost:8086 · Grafana at http://localhost:3000

07
Build the Node-RED flow
visual programming, no code needed

In Node-RED (localhost:1880), create this flow by dragging nodes:

  1. MQTT In node → topic: solar/sensors, server: localhost:1883
  2. JSON node → parses the incoming message string to an object
  3. Function node → adds a timestamp field
  4. InfluxDB Out node → measurement: sensor_readings, host: influxdb:8086

That's it. Every MQTT message from your ESP32 is now automatically stored in InfluxDB. Node-RED also has a built-in dashboard (install node-red-dashboard palette) for quick visualisation during testing.

5 · Phase 4 — AI decision controller

Phase 4 · estimated time: 1–2 weeks · this is the core of your project

A Python script that runs on a schedule, reads the latest sensor data + market price + weather forecast, applies decision logic, and publishes a command.

08
Install Python dependencies
one-time setup
pip install influxdb-client paho-mqtt requests schedule pulp
09
Build the decision controller — start with rules, upgrade to optimization
Python · ai_controller.py
"""
SolarAI Decision Controller — v1.0 (rule-based)
Runs every 15 minutes via schedule.
Reads: InfluxDB (sensors) + SMARD API (price) + OpenWeatherMap (forecast)
Writes: MQTT command topic
"""

import paho.mqtt.client as mqtt
import requests, schedule, time, json
from influxdb_client import InfluxDBClient
from datetime import datetime

# ── CONFIG ───────────────────────────────────
INFLUX_URL    = "http://localhost:8086"
INFLUX_TOKEN  = "your-influx-token"  # from InfluxDB UI
INFLUX_ORG    = "solarai"
INFLUX_BUCKET = "energy_data"
MQTT_BROKER   = "localhost"
MQTT_CMD      = "solar/command"
OWM_API_KEY   = "your-openweathermap-key"
LOCATION      = "Regensburg,DE"

# ── THRESHOLDS (match your Excel Sheet 4) ────
SELL_PRICE_MIN  = 0.28   # €/kWh
SELL_SOC_MIN    = 0.70   # 70% battery
STORE_SOC_MAX   = 0.80
STORE_THERM_MAX = 75     # °C
USE_PRICE_MAX   = 0.18
IMPORT_SOC_MIN  = 0.20

def get_latest_sensors():
    """Read latest sensor values from InfluxDB."""
    client = InfluxDBClient(url=INFLUX_URL, token=INFLUX_TOKEN, org=INFLUX_ORG)
    query = '''
    from(bucket: "energy_data")
      |> range(start: -5m)
      |> filter(fn: (r) => r._measurement == "sensor_readings")
      |> last()
    '''
    tables = client.query_api().query(query)
    sensors = {}
    for table in tables:
        for record in table.records:
            sensors[record.get_field()] = record.get_value()
    return sensors

def get_spot_price():
    """Fetch current electricity price from SMARD (Bundesnetzagentur)."""
    # SMARD API: https://www.smard.de/en/developer-tools
    # For prototype, use a simple approximation or cached value
    url = "https://api.smard.de/app/chart_data/4169/DE/index_quarterhour.json"
    try:
        r = requests.get(url, timeout=5)
        data = r.json()
        latest = data['timestamps'][-1]
        price_eur_mwh = data['series'][-1][1]
        return price_eur_mwh / 1000  # convert MWh → kWh
    except:
        return 0.25  # fallback if API down

def get_solar_forecast():
    """Get cloud cover for next 6h from OpenWeatherMap."""
    url = f"https://api.openweathermap.org/data/2.5/forecast?q={LOCATION}&appid={OWM_API_KEY}"
    try:
        r = requests.get(url, timeout=5)
        forecasts = r.json()['list'][:2]  # next 6h (2 x 3h slots)
        avg_cloud = sum(f['clouds']['all'] for f in forecasts) / len(forecasts)
        return avg_cloud  # 0–100, 100 = fully cloudy
    except:
        return 50  # fallback

def decide(sensors, spot_price, cloud_cover):
    """
    Rule-based decision engine.
    Returns: ('SELL'|'STORE'|'USE', reason_string)
    """
    soc   = sensors.get('battery_soc', 0.5)
    solar = sensors.get('solar_kw', 0)
    therm = sensors.get('thermal_temp', 60)
    load  = sensors.get('home_load_kw', 1.5)

    # Priority 1: Safety — import if battery critically low
    if soc < IMPORT_SOC_MIN:
        return "USE", "Battery critically low — use stored + import"

    # Priority 2: Sell — price is high and we have enough stored
    if spot_price > SELL_PRICE_MIN and soc > SELL_SOC_MIN:
        return "SELL", f"Price €{spot_price:.3f} > threshold, SOC {soc*100:.0f}%"

    # Priority 3: Store — solar surplus and storage not full
    if solar > load and soc < STORE_SOC_MAX:
        return "STORE", f"Solar surplus {solar-load:.1f}kW, charging battery"

    # Priority 4: Heat thermal store if solar available
    if solar > 0.5 and therm < STORE_THERM_MAX:
        return "STORE", f"Solar available, heating thermal to {STORE_THERM_MAX}°C"

    # Priority 5: Use stored — low price or cloudy forecast
    if spot_price < USE_PRICE_MAX or cloud_cover > 70:
        return "USE", "Low price or cloudy forecast — draw from storage"

    # Default: use what solar provides, no action
    return "USE", "Default — consuming available solar"

def run_controller():
    sensors    = get_latest_sensors()
    spot_price = get_spot_price()
    cloud_cover = get_solar_forecast()
    decision, reason = decide(sensors, spot_price, cloud_cover)

    payload = json.dumps({
        "decision":   decision,
        "reason":     reason,
        "price":      spot_price,
        "timestamp":  datetime.utcnow().isoformat()
    })

    client = mqtt.Client()
    client.connect(MQTT_BROKER, 1883)
    client.publish(MQTT_CMD, payload)
    client.disconnect()
    print(f"[{datetime.now().strftime('%H:%M')}] Decision: {decision} — {reason}")

# Run every 15 minutes
schedule.every(15).minutes.do(run_controller)
run_controller()  # run once immediately on start
while True:
    schedule.run_pending()
    time.sleep(1)
ℹ Upgrading to optimization later: Once the rule-based version works, replace the decide() function with a linear programming optimizer using PuLP. It will maximize revenue over a 24h horizon given forecast prices and solar predictions. This is the step that makes it a true AI controller.

7 · Phase 6 — External data sources

12
Connect to German electricity market prices (SMARD)
free · official Bundesnetzagentur data

SMARD (Strommarktdaten) is the official platform of the Federal Network Agency. It provides real-time and historical Day-Ahead and Intraday electricity prices in Germany at 15-minute resolution.

  • API documentation: smard.de/en/developer-tools
  • The API is completely free, no API key required
  • Data is updated every 15 minutes — perfect for your control cycle
  • Key endpoint: https://www.smard.de/app/chart_data/4169/DE/
13
Connect to solar irradiance forecast
OpenWeatherMap free tier · 1000 calls/day
  • Register at openweathermap.org/api → free tier is enough
  • Use the One Call API 3.0 — it gives hourly cloud cover, GHI (Global Horizontal Irradiance), and UV index for 48h ahead
  • For more precise solar forecasting later, look at Solcast (free tier for researchers) which gives direct solar irradiance forecasts on rooftop level

10 · Where to find everything

Hardware

  • Berrybase.de — best German source for ESP32, Raspberry Pi, and sensors. Faster shipping than AliExpress.
  • AZ-Delivery.de — good quality modules with German documentation and Arduino examples included.
  • Amazon.de — for anything time-sensitive with Prime delivery.

Learning resources

  • ESP32 + MQTT + Node-RED: Search "ESP32 MQTT Node-RED InfluxDB tutorial" on YouTube — Andreas Spiess channel has excellent videos on exactly this stack.
  • InfluxDB basics: docs.influxdata.com/influxdb/v2 — the official docs are very good.
  • PuLP optimization: "PuLP linear programming Python tutorial" — used for battery scheduling optimization.
  • Grafana dashboards: grafana.com/tutorials — the beginner tutorial with InfluxDB takes ~2 hours.

German-specific resources

  • SMARD API: smard.de/en/developer-tools — German electricity market data, completely free.
  • Bundesnetzagentur Monitoringbericht — good reference for German grid regulations and feed-in tariff (Einspeisevergütung) structure.
  • EEG 2023 (Erneuerbare-Energien-Gesetz) — the legal framework for solar feed-in in Germany. Important to understand before scaling up.
